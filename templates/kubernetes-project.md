## Naming Conventions

### CRDs and Custom Resources
- CRD group: use a domain you control (`myproject.example.com`)
- CRD names: `<plural>.<group>` (e.g., `inferenceservices.serving.example.com`)
- Kind: PascalCase, singular (`InferenceService`, not `inferenceservice`)
- API versions: `v1alpha1`, `v1beta1`, `v1`

### Labels and Annotations
Use the standard Kubernetes label prefixes (`app.kubernetes.io/name`, `app.kubernetes.io/version`, etc.). Custom labels should use your project domain: `myproject.example.com/environment: production`. Do not use unprefixed labels except `app` and `version` for backward compatibility.

### Resource Names
Lowercase with hyphens, under 63 characters, descriptive: `model-inference-server` not `server1`.

## Security

### Pod Security Standards

Every namespace should declare a Pod Security Standard. Use `restricted` as your default; drop to `baseline` only with a documented reason. Never use `privileged` unless you are building infrastructure-level components (CNI plugins, CSI drivers, log collectors).

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: my-app
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted
```

Key `restricted` requirements: `runAsNonRoot: true`, `allowPrivilegeEscalation: false`, `seccompProfile.type: RuntimeDefault`, drop all capabilities, read-only root filesystem where possible.

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1001
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  seccompProfile:
    type: RuntimeDefault
  capabilities:
    drop: ["ALL"]
```

### RBAC Least Privilege

- Define RBAC rules with `kubebuilder` markers in `config/rbac/`:
  ```go
  //+kubebuilder:rbac:groups=myproject.example.com,resources=myresources,verbs=get;list;watch;create;update;patch;delete
  //+kubebuilder:rbac:groups=myproject.example.com,resources=myresources/status,verbs=get;update;patch
  ```
- Never grant `cluster-admin` to an operator.
- Use `Role`/`RoleBinding` for namespace-scoped resources. Use `ClusterRole`/`ClusterRoleBinding` only for cluster-scoped objects or cross-namespace watches.
- Avoid wildcard verbs and resources. Enumerate explicitly.
- Audit RBAC periodically. Permissions accumulate and are rarely removed.
- Verify with `kubectl auth can-i --list --as=system:serviceaccount:<ns>:<sa>`.

### Network Policies

Default-deny ingress and egress per namespace, then add targeted allow rules:

```yaml
# Default deny all
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
spec:
  podSelector: {}
  policyTypes: [Ingress, Egress]
---
# Allow specific ingress
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-traffic
spec:
  podSelector:
    matchLabels:
      app.kubernetes.io/name: api-server
  policyTypes: [Ingress]
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: ingress-nginx
      ports:
        - protocol: TCP
          port: 8080
```

Important: multiple selectors in a single `from` list item create an AND condition. Separate list items create OR. Getting this wrong silently breaks your policy.

### Image Security and Supply Chain

Scan every image before it reaches a cluster. Block deployment of images with critical CVEs.

```bash
trivy image --severity HIGH,CRITICAL --exit-code 1 my-image:v1.0.0
cosign sign --key cosign.key my-registry.com/my-image:v1.0.0
cosign verify --key cosign.pub my-registry.com/my-image:v1.0.0
```

Use admission controllers to enforce image policies:
- **Kyverno** or **OPA Gatekeeper** for policy-as-code (require signatures, block `latest`, enforce registry allowlists)
- **Sigstore policy-controller** for signature verification at admission time

### Secrets Management

Do not store secrets in plain YAML committed to git. Options from most to least preferred:

1. **External Secrets Operator** with HashiCorp Vault, AWS Secrets Manager, or Azure Key Vault
2. **Sealed Secrets** (Bitnami) for encrypted secrets that live in git
3. **SOPS** with age or KMS for encrypting values in manifest files
4. **Vault Agent Injector** for sidecar-based injection at runtime

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: my-app-secrets
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: my-app-secrets
    creationPolicy: Owner
  data:
    - secretKey: database-password
      remoteRef:
        key: secret/data/my-app
        property: db_password
```

## Performance and Resource Management

### Resource Requests and Limits

Every container must have resource requests. Memory limits are required. CPU limits are debatable; omit them if throttling is a concern for latency-sensitive workloads.

```yaml
resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    memory: 256Mi  # CPU limit optional for latency-sensitive apps
```

When you omit CPU limits, set a ResourceQuota on the namespace to prevent runaway consumption.

### LimitRange and ResourceQuota

Set namespace-level defaults with LimitRange so pods without explicit resource specs get sane values. Enforce total consumption per namespace with ResourceQuota to prevent any single application from consuming the entire cluster:

```yaml
# LimitRange: per-container defaults
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
spec:
  limits:
    - default: {cpu: 500m, memory: 256Mi}
      defaultRequest: {cpu: 100m, memory: 128Mi}
      type: Container
---
# ResourceQuota: namespace totals
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
spec:
  hard:
    requests.cpu: "10"
    requests.memory: 20Gi
    limits.memory: 40Gi
    pods: "50"
```

### Horizontal Pod Autoscaler (HPA)

Use HPA v2 for custom metrics and fine-grained scaling behavior. The default scaling is often too aggressive for production.

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-app
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target: {type: Utilization, averageUtilization: 70}
    - type: Pods
      pods:
        metric: {name: http_requests_per_second}
        target: {type: AverageValue, averageValue: "1000"}
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies: [{type: Percent, value: 10, periodSeconds: 60}]
    scaleUp:
      stabilizationWindowSeconds: 30
      policies: [{type: Percent, value: 50, periodSeconds: 60}]
```

Key tuning points:
- `stabilizationWindowSeconds` for scale-down: at least 300s. Premature scale-down causes request spikes.
- Percentage-based scale-down prevents removing too many pods at once.
- Custom metrics require a reliable pipeline (Prometheus adapter or KEDA). Missing metrics halt scaling entirely.

### Vertical Pod Autoscaler (VPA)

Use VPA in `Off` mode for recommendations only, or `Initial` to set resources on pod creation. Do not run VPA and HPA on the same metric for the same deployment; they will fight each other.

### Pod Disruption Budgets (PDB)

Always set a PDB for production workloads. Without one, node drains during cluster upgrades can take down your entire application.

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: my-app-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app.kubernetes.io/name: my-app
```

Never set `minAvailable` equal to the replica count. That blocks all voluntary disruptions including node drains.

### Topology Spread and Affinity

Spread pods across failure domains. This is not optional for production.

```yaml
topologySpreadConstraints:
  - maxSkew: 1
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule
    labelSelector:
      matchLabels: {app.kubernetes.io/name: my-app}
  - maxSkew: 1
    topologyKey: kubernetes.io/hostname
    whenUnsatisfiable: ScheduleAnyway
    labelSelector:
      matchLabels: {app.kubernetes.io/name: my-app}
```

Use `DoNotSchedule` for zone spread (hard) and `ScheduleAnyway` for node spread (soft). This ensures zone-level fault tolerance without blocking scheduling when nodes are imbalanced.

## Networking

### Service Types

- **ClusterIP** (default): Internal-only service-to-service communication.
- **NodePort**: Static port on every node. Useful for dev; rarely appropriate for production.
- **LoadBalancer**: Provisions an external LB. Each service gets its own, which is expensive; prefer Ingress for HTTP.
- **ExternalName**: CNAME alias. Bypasses kube-proxy. Use sparingly.

### Headless Services

Use `clusterIP: None` when you need direct pod DNS resolution (StatefulSets, databases, custom discovery). Each pod gets `<pod-name>.<service>.<namespace>.svc.cluster.local`.

### DNS Patterns

- Services: `<service>.<namespace>.svc.cluster.local`
- Headless pods: `<pod-name>.<service>.<namespace>.svc.cluster.local`
- Use short names for intra-cluster calls. Use FQDN with trailing dot only when avoiding search domain appending.

### Ingress and Routes

Use a single Ingress controller (NGINX, HAProxy, Traefik) to multiplex HTTP traffic. Always set `ingressClassName`. On OpenShift, prefer Routes for native edge/reencrypt/passthrough TLS termination.

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  ingressClassName: nginx
  tls:
    - hosts: [my-app.example.com]
      secretName: my-app-tls
  rules:
    - host: my-app.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service: {name: my-app, port: {number: 8080}}
```

### Service Mesh

If you run Istio, Linkerd, or OpenShift Service Mesh:
- The sidecar proxy must start before your app container to intercept traffic.
- Init containers run before sidecars. If your init container needs in-mesh network access, it will fail. Use `holdApplicationUntilProxyStarts: true` (Istio) or restructure init logic.
- Exclude health check ports from mesh traffic if probes hit localhost.

## Helm Chart Conventions

- Follow [Helm best practices](https://helm.sh/docs/chart_best_practices/).
- Use `values.yaml` for all configurable parameters with sensible defaults and comments.
- Include `NOTES.txt` and `_helpers.tpl`.

- Include PDB, NetworkPolicy, and ServiceMonitor templates, toggleable via values.

## Operator Development

### Controller Best Practices
- Controllers must be idempotent. Multiple reconciliations with the same input produce the same result.
- Use `controller-runtime`'s built-in retry and rate-limiting. No custom retry loops.
- Add finalizers when your controller creates external resources that need cleanup.
- Use status conditions to communicate resource state (`Ready`, `Degraded`, `Progressing`).
- Use exponential backoff for external API calls.

### Error Handling
- `ctrl.Result{RequeueAfter: time.Minute}` for transient errors.
- `ctrl.Result{}` when reconciliation is complete.
- Structured logging with namespace and name on every log line.
- Never panic in controllers.

### Finalizer Pattern

```go
const finalizerName = "myproject.example.com/cleanup"

func (r *MyResourceReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    var resource myv1.MyResource
    if err := r.Get(ctx, req.NamespacedName, &resource); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)
    }
    if resource.DeletionTimestamp.IsZero() {
        if !controllerutil.ContainsFinalizer(&resource, finalizerName) {
            controllerutil.AddFinalizer(&resource, finalizerName)
            return ctrl.Result{}, r.Update(ctx, &resource)
        }
    } else {
        if controllerutil.ContainsFinalizer(&resource, finalizerName) {
            if err := r.cleanupExternalResources(ctx, &resource); err != nil {
                return ctrl.Result{}, err
            }
            controllerutil.RemoveFinalizer(&resource, finalizerName)
            return ctrl.Result{}, r.Update(ctx, &resource)
        }
        return ctrl.Result{}, nil
    }
    // Normal reconciliation
    return ctrl.Result{}, nil
}
```

```yaml

  name: my-app-alerts

    - name: my-app
```

### Structured Logging

All containers should emit structured JSON logs to stdout. Never write log files inside the container.

```go
logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{Level: slog.LevelInfo}))
logger.Info("reconciliation complete",
    "resource", req.NamespacedName,
    "duration_ms", elapsed.Milliseconds(),
)
```

Rules: include correlation IDs (trace ID, request ID) in every log line. Use consistent field names across services. Control log level via environment variable or ConfigMap.

### Distributed Tracing

Use OpenTelemetry for request tracing across services. Deploy the OTel Collector as a sidecar or DaemonSet. For Go controllers, use `go.opentelemetry.io/otel` to trace reconciliation loops.

## GitOps

### ArgoCD Application Patterns

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app-production
  namespace: argocd
spec:
  project: my-team
  source:
    repoURL: https://github.com/my-org/my-app.git
    targetRevision: main
    path: deploy/kustomize/overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: my-app-production
  syncPolicy:
    automated: {prune: true, selfHeal: true}
    syncOptions: [CreateNamespace=true, ServerSideApply=true]
    retry:
      limit: 5
      backoff: {duration: 5s, factor: 2, maxDuration: 3m}
```

- `selfHeal: true` reverts manual cluster changes automatically.
- `ServerSideApply=true` avoids field ownership conflicts.
- `prune: true` deletes resources removed from git. Test in staging first.
- Use AppProjects to restrict which clusters and namespaces a team can deploy to.

### Kustomize Overlay Strategy

The base contains the full application. Overlays contain only differences. Production overlays should add PDB, HPA, and tighter resource limits.

### Helm vs Kustomize

Choose **Helm** when distributing to external users, needing templating logic, or publishing to a chart repository. Choose **Kustomize** when managing internal environments with mostly static manifests and small per-environment overrides. Many teams use both: Helm for packaging, Kustomize for environment overlays on rendered output.

### Drift Detection

- ArgoCD shows drift per application. Alert on `OutOfSync` status persisting more than a few minutes.
- Run `kubectl diff -f <manifests>` periodically against production to catch unmanaged changes.

## OpenShift-Specific Notes

- Routes are preferred over Ingress. They support edge/reencrypt/passthrough TLS natively.
- Security Context Constraints (SCCs) restrict pod capabilities. Test with `restricted-v2` SCC.
- Use the built-in image registry for development: `image-registry.openshift-image-registry.svc:5000`

## Troubleshooting

### kubectl debug

Use ephemeral debug containers for live troubleshooting without modifying the pod spec:

```bash
kubectl debug -it my-pod --image=registry.access.redhat.com/ubi9/toolbox --target=my-container
kubectl debug node/my-node -it --image=registry.access.redhat.com/ubi9/toolbox
kubectl debug my-pod -it --copy-to=my-pod-debug --container=my-container -- /bin/sh
```

### CrashLoopBackOff Diagnosis

The container keeps exiting. Backoff increases with each restart (10s, 20s, 40s, up to 5 minutes).

```bash
kubectl describe pod <pod-name> | grep -A 5 "Last State"
kubectl logs <pod-name> --previous
# Exit 1:   Application error. Check logs.
# Exit 137: OOMKilled (SIGKILL). Increase memory limit.
# Exit 139: Segfault (SIGSEGV). Debug the binary.
# Exit 143: SIGTERM. Check liveness probe misconfiguration.
```

### ImagePullBackOff Debugging

```bash
kubectl describe pod <pod-name> | grep -A 10 Events
# Common causes: image does not exist, auth failure (check imagePullSecrets),
# registry rate limiting, network policy blocking egress to registry.
```

### OOMKilled Analysis

Exit code 137 means the kernel OOM killer terminated the container for exceeding its memory limit.

```bash
kubectl describe pod <pod-name> | grep -i oom
kubectl top pod <pod-name> --containers
```

Fix: increase the limit if too low, profile for memory leaks (Go: pprof, Java: heap dumps, Python: tracemalloc), or split the workload. Check VPA recommendations.

### Pod Lifecycle Events

1. **Scheduling**: assigned to node. Failures show as `Pending` (insufficient resources, taints, affinity).
2. **Image Pull**: failures show as `ImagePullBackOff` or `ErrImagePull`.
3. **Init Containers**: run sequentially. Failure restarts the pod.
4. **Container Start**: main containers start. `postStart` hooks run (not guaranteed before entrypoint).
5. **Readiness Probe**: once passing, pod joins Service endpoints.
6. **Liveness Probe**: ongoing. Failure restarts the container. Different from readiness.
7. **PreStop Hook**: runs before SIGTERM. Use for graceful drain.
8. **SIGTERM/SIGKILL**: SIGTERM sent to PID 1, then SIGKILL after `terminationGracePeriodSeconds` (default 30s).

## Common Pitfalls

### Resource Limits
- **Missing memory limits cause node-level OOMKills.** Without limits, one pod can consume all node memory and the kernel kills victims across the entire node. Always set memory limits.
- **CPU limits cause throttling.** For latency-sensitive workloads (API servers, inference), consider CPU requests without limits. CFS quota enforcement throttles even when spare CPU exists.

### Probes
- **Readiness vs liveness confusion.** Liveness restarts the container. Readiness removes it from endpoints. For slow-starting apps, use a startup probe. Do not use a short-timeout liveness probe for startup; it kills the container before it finishes.
- **Liveness probes that check dependencies.** If liveness checks the database, a DB outage restarts all pods, making recovery worse. Liveness should check only process health. Use readiness for dependency checks.

```yaml
startupProbe:
  httpGet: {path: /healthz, port: 8080}
  failureThreshold: 30
  periodSeconds: 10
livenessProbe:
  httpGet: {path: /healthz, port: 8080}
  periodSeconds: 15
readinessProbe:
  httpGet: {path: /readyz, port: 8080}
  periodSeconds: 5
```

### Rolling Updates
Default `maxUnavailable: 25%` can take down all pods for small replica counts. Set explicitly:
```yaml
strategy:
  type: RollingUpdate
  rollingUpdate: {maxUnavailable: 0, maxSurge: 1}
```
Tradeoff: slower rollouts, but no downtime.

### PVC Access Modes
- `ReadWriteOnce` (RWO): mounted read-write on a single **node**, not a single pod. Multiple pods on the same node share it.
- `ReadWriteMany` (RWX): needed for cross-node shared storage. NFS, CephFS, EFS support it. EBS does not.
- `ReadWriteOncePod` (RWOP): true single-pod exclusive access (Kubernetes 1.27+).

### DNS Resolution
- If DNS fails, check CoreDNS pods in `kube-system`.
- Default `ndots:5` means short names generate 5 search domain lookups before trying the name as-is. For apps making many external DNS queries, reduce it:
  ```yaml
  dnsConfig:
    options: [{name: ndots, value: "2"}]
  ```

### ConfigMap and Secret Reloads
- Mounted ConfigMaps/Secrets update eventually (kubelet sync, ~60 seconds), but the app must detect and reload. Not atomic.
- Env vars from ConfigMaps/Secrets are NOT updated on the running pod. Pod restart required.
- Use stakater/Reloader for automatic rolling restarts, or use a checksum annotation in the Deployment template:
  ```yaml
  annotations:
    checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}
  ```

### Init Container Ordering
- Init containers run sequentially before main containers and sidecars. If an init container needs in-mesh network access, it will fail because the sidecar is not running yet.
- Use init containers for: database migrations, config generation, waiting for dependencies, populating shared volumes.

### Other Common Mistakes
- Do not use `kubectl` in automation without checking exit codes.
- Do not assume a namespace exists.
- Do not hardcode cluster-specific values (domain names, registry URLs) in manifests.
- Do not use em dashes in documentation or comments.
- Do not commit kubeconfig files or service account tokens to git.

## Multi-Cluster Operations

### Cluster API

Use Cluster API (CAPI) for declarative cluster lifecycle management. Define clusters as Kubernetes resources with infrastructure-specific providers (AWS, Azure, vSphere, bare metal).

### Multi-Cluster Deployment Patterns

- **Hub and spoke**: One management cluster runs ArgoCD/Flux and deploys to workload clusters. Most common pattern.
- **Replicated control planes**: Each cluster is independent, syncing from the same git repo but different overlay paths.
- **Federation (Admiralty, Liqo)**: Pods scheduled in one cluster transparently dispatched to another. Use for burst capacity.

### Workload Identity Across Clusters

- **SPIFFE/SPIRE** for cross-cluster workload identity with mutual authentication.
- **Cloud IAM integration**: IRSA (EKS), Workload Identity (GKE) to bind ServiceAccounts to cloud IAM roles without static credentials.
- **Cross-cluster service communication**: multi-cluster service mesh (Istio multi-primary, Linkerd multi-cluster) or DNS-based discovery with ExternalDNS.

### Multi-Cluster Observability

- Centralized Prometheus with remote-write, or Thanos/Cortex/Mimir for aggregation.
- Consistent labels across clusters (`cluster` label on all metrics).
- Central log store (Loki, Elasticsearch) with cluster identification in every log line.
